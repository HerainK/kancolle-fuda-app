# 艦これ 出撃制限札管理アプリ 開発指示書 (v3)

## 1. 背景・目的

「艦隊これくしょん(艦これ)」の期間限定海域(イベント)には「出撃制限札(通称:札)」というシステムがある。

- イベント海域(出撃ポイント)ごとに「出撃を許可する札」が設定されている。
- 札が一切ついていない艦娘は、どの海域にも制限なく出撃できる。
- ある海域に出撃すると、その艦娘に「その海域(または出撃地点)に対応する札」が付与される。一度ついた札は外せない。
- 複数の海域が同じ札を共有していることがあり、その場合はその札を持つ艦娘はどちらの海域にも出撃できる。
- 出撃制限が発生するのは主に難易度「甲」「乙」。「丙」「丁」では札自体は付与されるが制限は無視できる。

このため提督は、攻略サイト等の情報を元に「どの艦をどの札(海域)に割り振るか」を事前に計画する必要がある。

**本アプリの位置づけ**: 実際のゲーム内出撃を記録するものではなく、出撃前に「艦娘を箱に仕分けする」ことで割り振り計画を立てるための**プランニングツール**である。そのため、実際のゲームとは異なり、アプリ上の割り当ては自由に編集・やり直しができてよい(箱から外せば札も解除される)。

**将来的な公開について**: 現時点ではユーザー個人利用が前提だが、将来的に静的ホスティング(GitHub Pages / Vercel / Netlify 等)で一般公開し、他の提督にも使ってもらう可能性がある。そのため実装時は、特定ユーザー個人を前提としたハードコード(ユーザー名の決め打ち、個人の艦娘データを初期値に埋め込む、等)を避け、初期状態は空のデータから始まる汎用的な作りにしておくこと。艦娘マスタ以外のデータ(保有艦娘・イベント・箱など)は、公開後も引き続き各ユーザーのブラウザのlocalStorageに閉じた個人データとして扱う設計を維持する(サーバー側にユーザーデータを集約する必要はない)。

## 2. 用語定義

| 用語 | 説明 |
|---|---|
| 艦娘マスタ | ゲームに実装されている艦娘の名前・艦種・姉妹艦(艦級)・改装形態の一覧 |
| 保有艦娘(艦娘インスタンス) | ユーザーが実際に保有する艦娘の個別データ。同じ艦娘を複数保有可能(それぞれ別インスタンス) |
| イベント | 期間限定海域全体。開始のたびに新規作成する |
| 札(タグ) | イベントごとに用意する出撃制限の識別子(例:「第三十一戦隊」)。艦娘が実際に保持するのはこの「札」であり、箱ではない |
| 箱 | 艦娘を仕分ける入れ物。「艦隊の種類」「札」「用途(名前)」の3つを持つ。1つのイベント内に複数作成できる |
| 艦隊の種類 | 箱に設定する属性。「通常」「遊撃」「連合」の3種類から選択する。箱に入れられる艦娘の最大数を決める |
| 未タグ | まだどの箱にも入ったことがなく、札を持っていない艦娘の状態。どの箱にも自由に配置できる |

## 3. 機能要件

### 3.1 艦娘マスタデータ管理
- 艦これに実装されている全艦娘を対象に、以下を持つマスタデータを用意する
  - 艦娘名 / 艦種(戦艦・空母・重巡・軽巡・駆逐艦・潜水艦 等) / 姉妹艦(艦級)グループ / 改装形態リスト(艦娘ごとに異なる。無印/改/改二/改二乙/改二丙 等)
- マスタデータは初期構築時に一括で用意する(詳細は「7. マスタデータの用意方法」を参照)
- **メンテナンス画面**を用意し、新艦娘の追加・既存艦娘への改装形態追加・艦種や艦級の修正を後から手動で行えるようにする

### 3.2 保有艦娘登録(艦娘インスタンス)

**メイン登録方式: マスタ一覧からのクリック登録**
- 艦娘マスタの全艦娘を、艦種→艦級(姉妹艦)ごとに整理した一覧(グリッドまたはリスト)として1画面に表示する
- 検索・絞り込み(名前検索、艦種フィルタ)でこの一覧を絞り込める
- 未保有の艦娘をタップ/クリックすると、その場でレベル・改装形態(その艦娘に存在する形態のみ選択肢に出す)を入力する小さなフォームが開き、確定すると艦娘インスタンスとして登録される
- 既に保有している艦娘は、一覧上で「保有済み」のバッジや保有数(例:「×2」)を表示する。保有済みの艦娘を再度タップすると、
  - 新規インスタンスとして追加登録(2隻目以降の重複所持に対応)
  - または既存インスタンスのレベル・改装形態を編集
  のどちらかを選べるようにする
- この方式により、艦種選択→名前選択→レベル入力→改装形態選択→登録ボタンを艦娘ごとに毎回繰り返す必要がなくなり、一覧を見ながら手早く登録できる

**補助登録方式: フリーテキスト一括貼り付け(パワーユーザー向け任意機能)**
- 上記のクリック登録とは別に、テキストエリアに1行1隻で「艦娘名 レベル [改装形態の略称]」の形式(例:「雪風 120 改二」)を貼り付け、まとめて解析・一括登録できる機能も用意する
- 名前はあいまい一致でマスタと照合し、改装形態が省略された場合は「無印」扱いとする
- 解析後、マッチしなかった行や複数候補があって判定できない行は確認画面でハイライトし、手動修正できるようにする
- 大量の艦娘を一気に登録したい場合など、クリック登録が面倒に感じるユーザー向けの代替手段として位置づける(必須機能ではなく、フェーズ4以降の任意拡張として実装してよい)

**共通仕様**
- 同じ艦娘を複数登録可能(例: 雪風を2隻所持している場合、それぞれ別インスタンスとして管理し、レベル・改装形態・箱への所属状況も個別に持つ)
- 登録済み艦娘の一覧(艦種→艦級ごとに整理)から、いつでもレベル・改装形態を編集できる

### 3.3 イベント・札・箱管理
1. イベントを新規作成する(イベント名、開始日、終了日)
2. イベントに対して、そのイベントで使用する**札(タグ)の一覧**を登録する(例:「第三十一戦隊」「北方海域」など。開発側=ユーザー自身が攻略情報を元に事前に用意する)
3. 登録した札を使って**箱**を作成する。箱には以下を設定する
   - 艦隊の種類(通常・遊撃・連合のいずれか)
   - 札(2.で登録した札一覧から選択)
   - 用途(自由記述の名前。例:「E1-1ゲージ目」「第三十一戦隊」)
- 「札管理画面」は現在進行中のイベントを表示する(イベントを切り替えると表示される箱も切り替わる。過去イベントのデータは削除せず、イベント一覧から呼び出して参照できるようにする)

### 3.4 割り当て画面(ドラッグ&ドロップ)
- 画面には「未タグの艦娘プール」と、イベント内の「箱」一覧を並べて表示する
- 艦娘をドラッグして箱にドロップした際の挙動は以下の通り
  1. **艦娘が未タグの場合**: そのままドロップを受け入れ、箱の「札」がその艦娘に付与される(以後その艦娘は同じ札の箱にのみ入れられるようになる)
  2. **艦娘が既にタグを持っており、ドロップ先の箱の札と一致する場合**: ドロップを受け入れる。この艦娘は既に他の箱に入っていてもよく、**同じ札の箱であれば複数の箱に同時に所属できる**(例: 「通常艦隊・E1-1ゲージ目・第三十一戦隊」の箱に入れた艦娘を、同じ「第三十一戦隊」の札を持つ「遊撃艦隊・E2-ギミック・第三十一戦隊」の箱にも重ねて入れられる)
  3. **艦娘が既にタグを持っており、ドロップ先の箱の札と一致しない場合**: **ドロップを拒否する**(強めのブロック。例外は設けない)
- **箱の最大収容数は艦隊の種類で固定でハードチェックする**: 通常=6隻、遊撃=7隻、連合=12隻。上限に達している箱へのドロップは拒否する
- 艦娘を箱から取り除く操作(ドラッグで戻す、またはボタン)も自由にできる。**ある艦娘がどの箱にも所属しなくなった場合、その艦娘のタグは自動的に解除され、再び未タグ状態(＝どの箱にも配置可能)に戻る**
- スマホでの操作性に配慮し、タップで「艦娘を選択→対象の箱を選んで配置/解除」という代替操作も用意する

### 3.5 ダッシュボード
- イベント単位で、箱ごとの現在の収容数(例:「4/6」)を一覧表示
- 艦娘ごとの現在のタグ状態(未タグ / 保持している札名)と、所属している箱の一覧を表示
- 未タグの艦娘数などのサマリーを表示

### 3.6 データの保存・共有
- ブラウザの localStorage に自動保存(サーバー不要)
- 艦娘マスタ・保有艦娘・イベント・札・箱・所属状況をすべて含めた JSON としてエクスポート/インポートできる
- 過去イベントのデータもイベント一覧から参照できるよう保持しておく(自動削除しない)

## 4. データモデル(概略)

```
ShipMasterEntry {
  id: string
  name: string
  shipType: string
  shipClass?: string
  refitForms: RefitForm[]
}

RefitForm {
  id: string
  name: string
}

ShipInstance {
  id: string
  masterId: string
  level: number
  refitFormId: string
  currentTagId: string | null   // 現在保持している札。未タグならnull
  memo?: string
}

Event {
  id: string
  name: string
  startDate: string | null
  endDate: string | null
  tags: Tag[]
  boxes: Box[]
}

Tag {
  id: string
  name: string   // 例: "第三十一戦隊"
}

Box {
  id: string
  fleetType: "normal" | "strike" | "combined"  // 通常・遊撃・連合
  tagId: string           // Tag.id への参照
  purpose: string         // 用途(自由記述)
  shipInstanceIds: string[]  // この箱に所属する艦娘インスタンスのID一覧
}
```

**制約ロジックまとめ**:
- `fleetType` ごとの最大収容数: normal=6, strike=7, combined=12(定数として固定)
- ドロップ受入判定: `ship.currentTagId === null || ship.currentTagId === box.tagId` かつ `box.shipInstanceIds.length < 最大収容数` の場合のみ許可
- 艦娘を箱から除去した際、その艦娘がどの `box.shipInstanceIds` にも含まれなくなったら `ship.currentTagId = null` に戻す

## 5. 画面構成

1. **艦娘マスタ管理画面**(メンテナンス用): マスタの一覧・新規追加・改装形態追加・編集
2. **保有艦娘登録画面**: 艦種フィルタ→名前選択→レベル・改装形態入力で登録。登録済み艦娘の一覧・編集
3. **イベント一覧画面**: イベントの新規作成・選択・過去イベント参照
4. **イベント設定画面**: 札(タグ)一覧の登録、箱(艦隊の種類・札・用途)の登録・編集
5. **割り当て画面(メイン画面)**: 未タグの艦娘プールと箱一覧を並べたドラッグ&ドロップUI。スマホはタップ選択方式に切り替え
6. **ダッシュボード画面**: 箱ごとの収容状況、艦娘のタグ状態一覧
7. **設定画面**: JSONエクスポート/インポート、データリセット

## 6. 非機能要件・技術スタック

- **形態**: Webアプリ(ブラウザ動作)。PC・スマートフォン両対応のレスポンシブデザイン
- **推奨スタック**: React + TypeScript + Vite、Tailwind CSS
- **ドラッグ&ドロップ**: `dnd-kit` の利用を推奨(タッチ操作対応のため)。スマホでは前述の通りタップ選択方式を代替手段として実装する
- **永続化**: localStorage(自動保存) + JSON手動エクスポート/インポート。バックエンドサーバーは不要
- **オフライン動作**: 通信不要で完結する構成(静的ホスティングで配信可能)
- **公開時の留意点**: 将来的に一般公開する可能性があるため、艦娘マスタデータは名前・艦種・改装形態名などのテキスト情報のみを扱い、公式イラストや立ち絵などの画像アセットは使用・同梱しないこと

## 7. マスタデータの用意方法(重要・最初に着手すべきタスク)

艦これの実装艦は300隻近くあり、それぞれ改装形態が異なるため、全艦分を人力で1件ずつ入力するのは非常に工数が大きい。以下の方針でClaude Codeに着手させる。

1. まず、艦娘マスタ(名前・艦種・艦級・改装形態)を網羅的に収録した、構造化されたデータソースの入手を試みる(有志が公開しているオープンデータ等)。入手できた場合は、それを本アプリのデータ形式(`ShipMasterEntry[]`)に変換するスクリプトを作成し、初期マスタデータを一括生成する
2. 構造化データソースが入手できない、または情報が古い/不完全な場合は、**主要な艦種ごとによく使われる艦娘を中心に少数から着手し**、メンテナンス画面から随時追加していくハイブリッド方式に切り替える。この場合は着手前にその旨をユーザーに報告し、初期収録範囲について確認を取ること
3. どちらの方式で着手するにせよ、新艦娘・新改装形態が実装された際に**ユーザー自身がメンテナンス画面から追加・修正できる**ことを前提に設計する(マスタデータの完全性を初回リリースの必須条件にしない)

## 8. 開発の進め方(フェーズ分け)

1. **フェーズ0**: 艦娘マスタデータの調査・収集・初期データ生成(上記7章の方針に従う)
2. **フェーズ1**: プロジェクト雛形作成(Vite + React + TS + Tailwind)、データモデルの型定義、localStorage保存/読込ユーティリティ、マスタデータの組み込み
3. **フェーズ2**: 艦娘マスタ管理画面(メンテナンス用)、保有艦娘登録画面(マスタ一覧からのクリック登録をメインで実装。フリーテキスト一括貼り付けは任意で、余裕があれば同フェーズ内、なければ後回しでよい)
4. **フェーズ3**: イベント・札・箱管理画面
5. **フェーズ4**: 割り当て画面(ドラッグ&ドロップ、タグ一致判定、艦隊種別ごとの最大収容数チェック、スマホ用タップ方式含む)ーコア機能
6. **フェーズ5**: ダッシュボード画面
7. **フェーズ6**: JSONエクスポート/インポート、UI/UXの仕上げ(レスポンシブ対応、警告・エラー表示のブラッシュアップ)
